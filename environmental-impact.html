Environmental impact of this site

What going static saves

← back to the calculator

TL;DR

Going static cuts this site's footprint by over 99%.

The Green Algorithms calculator used to be a Plotly Dash application: a Python server that rendered the page, shipped ~2.65 MB of JavaScript, and recomputed the results on the server every time an input changed. This version is a static site: ~31 kB transferred in total, all calculations run in your browser, and no server needs to stay on 24/7. The methodology below follows the spirit of Starlight's environmental impact page.

That is a ~86× reduction (−99%) in data transferred for a first visit. On top of that, every input change in the Dash app triggered a round trip to the server (a POST /_dash-update-component per callback); here the same calculation is a few microseconds of arithmetic in your browser, with zero network traffic.

2. Emissions from data transfer

Using the Sustainable Web Design model (the model behind websitecarbon.com): 0.81 kWh per GB transferred, at a global average carbon intensity of 442 gCO2e/kWh → ≈ 358 gCO2e per GB.

Per visit the static site saves ≈ 0.90 g CO2e. Small individually — but it scales with every visit, and it compounds with the server saving below.

2½. Estimate it for your own traffic

3. The always-on server

A static site is served from a CDN cache whose marginal cost per extra site is close to zero. The Dash app, in contrast, needed a dedicated dyno running Python/gunicorn around the clock — even at 4am with zero visitors.

We can estimate that server's footprint with the Green Algorithms calculator itself: one always-on average CPU core (7.46 W/core at full usage) plus 0.5 GB of memory (0.19 W), with a cloud PUE of 1.2 and 2024 US life-cycle carbon intensity (400 gCO2e/kWh):

1.2 × (7.46 W + 0.19 W) × 24 h × 365 = 80.4 kWh/year → ≈ 32 kg CO2e/year

(check this estimate in the calculator — it runs entirely in your browser.) This is conservative: it ignores redundant dynos, databases, staging environments and the idle inefficiency of a server doing nothing most of the time.

4. Total annual savings

For a site with 10,000 visits per month:

Source of savingsCO2e saved per year
Data transfer (10k visits × 12 months × 0.90 g)≈ 107 kg
Always-on server eliminated≈ 32 kg
Total≈ 139 kg CO2e/year

In this calculator's own reference units, 139 kg CO2e/year is equivalent to:

  • ≈ 13 tree-years of carbon sequestration,
  • ≈ 790 km driven in an average European passenger car,
  • ≈ 3 Paris–London flights.

For one small scientific web tool. The web serves billions of pages a day — defaulting to static delivery wherever interactivity doesn't truly require a server is one of the cheapest carbon optimisations there is.

Methodology notes & caveats

  • Transfer sizes are compressed (over-the-wire) bytes. The Dash figure was measured against the live app on 12 June 2026 and includes the lazily loaded Plotly bundle, which the page always fetches to draw its charts. Returning visitors with warm caches transfer less in both cases.
  • The Sustainable Web Design model is intentionally broad: it amortises data centres, networks and end-user devices into a single per-GB figure. Treat results as an order of magnitude, not a measurement.
  • The server estimate assumes a single small always-on instance. Real deployments (redundancy, monitoring, CI) usually add to this, so the saving is likely understated.
  • Static hosting isn't free of impact: CDN nodes also use energy. But the marginal share attributable to one ~31 kB site is negligible compared to a dedicated dyno.
